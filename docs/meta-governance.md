# 元规则:谁来管"改环境的人"(META-GOVERNANCE)

状态:环境宪法条款。作者:Claude+用户,2026-07-01。
关联:[[codex-agent-lab-purpose]]、`docs/proposal-subagent-gates.md`、
`docs/codex-claude-collaboration-protocol.md`。

## 一、被点破的悖论(用户 2026-07-01)

这个环境所有的治理(规则继承、fail-closed 安全门、任务三关卡)管的是
**"在环境里干活的 agent"**。但 Claude 和 Codex **修改环境本身**时,是
**站在环境之上的操作者** —— 我们给自己建的门,自己可以绕过。

证据(2026-07-01):今天十几次"改环境"的动作(建 run-record seam、
task-state kernel、merge-queue kernel……)几乎全部**没有进 tasks.json**,
因此**不受任务三关卡约束**。我们留了 run-record(有痕迹),但那是自愿的,
**没有任何门强制"改环境前必须走流程"**。改环境时,我们是裸奔的。

这不是设计缺陷,是**所有自我治理系统的根本极限**:一个系统无法完全用
自己约束"自己修改自己"这个动作。哲学上叫"谁来看守看守者"(quis
custodiet ipsos custodes)。

## 二、为什么不造"第四道门"

给"改环境"再加一道自动门,那道门**也是我们建的** —— 改那道门时我们又站在
它上面,于是需要第五道、第六道……**无限递归,没有终点**。而且这直接违背本
环境"私人工作台 + 轻量快走"的定位([[codex-agent-lab-purpose]]、
[[lightweight-fast-by-default]])。造无限套娃的门是过度工程。

## 三、解法:承认极限,人在顶层封顶

递归必须在某一层停下。**它停在"人"这一层。** 用户是这个治理链的终点、
最终裁判 —— 不是又一道自动门,而是**清醒地承认:机器无法约束"改机器本身",
所以由人来封顶。**

## 四、改环境的三条铁律(META RULES,写明的宪法,非自动门)

当 Claude 或 Codex **修改环境本身**(root 层:scripts/、lab_agents/、docs/、
registry/ 治理面、协议、门本身)时,必须遵守:

1. **留痕**:每次改环境必须产出 run-record(谁、改了什么、验证了什么、
   HEAD 前后 = 可回滚依据)。这已是习惯,现在是铁律。
2. **对方复查**:改环境的一方不能自我批准;必须由另一 lane 独立复查
   (读代码+跑门),或经用户确认。今天已在这么做(出题/答题/复查分离)。
3. **人封顶**:任何"改门本身 / 改这三条铁律 / 改协议"的动作,是**递归的顶点**,
   必须经用户明确同意。机器不得自行修改约束机器的元规则。

这三条不靠自动门强制(那会掉进无限递归),靠**写明 + 双方自觉 + 用户封顶**。
它们是"宪法",违反不会被程序拦下,但会被对方复查和用户发现 —— 这正是
"人在顶层"的含义。

## 五、诚实边界

- 这三条不能被程序 100% 强制(强制它们的程序又需要被强制……)。它们的效力来自
  透明(留痕可查)+ 互查(对方能发现)+ 人的最终权威。
- 这不是漏洞被"补上"了,是悖论被**如实承认并妥善安放** —— 把无法自动化的那部分,
  明确交给人。对私人工作台,这是正确且够用的解法。
- 若定位改为对外/生产级,元治理需要更强的外部审计(独立于两个 lane 的第三方),
  但那是另一个场景的事。
