关联:[[rule-inheritance]]、`docs/rule-inheritance.md`、`scripts/check-rule-ladder`、
`lab_agents/task_state.py`、`lab_agents/worktree_merge_queue.py`。

## 一、要解决的三个问题(你提出的)

1. **规则真读了吗**:结构上三层规则链已实测连通(底层→顶层4条路径全可达,
   `check-rule-ladder` 断链数=0)。但"文件可达"≠"Codex/子agent运行时真把上层
   规则读进了上下文"。你的原始担心:很久前让Codex重读协议,它可能只读了最小
   环境内部,顶层/中层到底读没读,无人能证。
2. **自动调度缺失**:今天7次Codex调用是我手动一个个串着派的(派→等→复查→再派)。
   高强度并发时,没有一个自动的东西去分配任务、排先后、防撞车。
3. **自动复查缺失**:今天每次Codex干完我都独立复查(跑测试+验文件)。同时跑十个
   时,我来不及逐个认真复查——复查会变敷衍(危险)或成为新瓶颈。

## 二、关键洞察:三个问题同一个根

它们其实是同一件事的三个面 —— **子agent/任务缺少一个强制的"开工门禁"和"收工闸"**:

- 规则没读 = 开工门禁没检查"你读全规则了吗"
- 调度靠手 = 没有门禁自动决定"谁现在能开工"
- 复查靠手 = 没有收工闸自动拦住"没验证过的产出"

所以不做三个零散功能,做**一个统一的任务生命周期闸**:每个任务从"排队"到
"完成",必须依次通过 3 道自动关卡。已有的 `task_state.py`(状态机)和
`worktree_merge_queue.py`(防撞车)是现成地基,这个方案是给它们加"关卡"。

## 三、统一设计:任务生命周期的 3 道关卡

每个派给子agent/Codex的任务,状态机里加三道自动关卡(fail-closed:不过就卡住):

**关卡1 — 开工门禁(解决"规则真读了")**
- 任务从 `pending` → `running` 前,执行方必须在任务记录里**声明它加载了哪些规则层**
  (顶层 AGENTS/CLAUDE、中层 workspace、底层 package、相关协议)。
- 一个新校验器 `check-rule-acknowledgment` 核对:该任务所在层级**要求**的规则层,
  是否都在声明里,且声明的文件路径**真实可达**(复用 check-rule-ladder 的路径解析)。
- 缺任何一层 → 任务卡在 `pending`,不准 `running`。把"读没读"从自觉变硬门。
- 注意:这证明的是"执行方声明+文件可达",仍不能100%证明模型真读进脑子;但它把
  "无声明就开工"堵死了,比现在强得多。这是诚实的边界。

**关卡2 — 自动派活(解决"自动调度")**
- 已有 `next_runnable_task()` 会算"哪个任务依赖已满足、可开工"。缺的是**自动执行**。
- 加一个 `scripts/dispatch-next`:读队列 → 取下一个可运行任务 → 检查关卡1 → 通过就
  自动 `omx-api exec` 派给 Codex(或标记归 Claude)→ 回写状态。
- **并行度闸**(你说烧钱不用管,但撞车要管):dispatch 一次最多放行 N 个,且每个占用
  独立 worktree(复用合并队列),避免同时改同一文件。N 先设保守值(如 2-3)。

**关卡3 — 收工闸(解决"自动复查")**
- 任务 `running` → `done` 前,必须先过 `verified`。加一个 `scripts/verify-task`:
  自动跑该任务声明的验证(测试、check-lab、目标文件断言),全绿才准进 `verified`。
- 红 → 任务打回 `running` 或标 `blocked`,**绝不自动进 done**。
- 这不是取代我的人工复查,是**过滤器**:机器先自动挡掉明显失败的,我只对"机器说通过
  的"做抽查/深查。把我从"逐个跑测试"里解放,盯真正需要判断的地方。

## 四、落地顺序(建议,轻量优先)

1. 关卡3(收工闸)先做 —— 价值最高、最独立,立刻能减我复查负担。
2. 关卡1(开工门禁)次之 —— 直接回答你"规则真读了吗"。
3. 关卡2(自动派活)最后 —— 风险最高(自动烧额度+自动改文件),要最谨慎,
   且必须先有 1 和 3 兜底才敢自动跑。

每步都:Claude 设计+建校验器,Codex 实现重活,Claude 独立复查,轻量不堆繁琐。

## 五、诚实的限制

- 关卡1 证明"声明了+可达",不证明"模型真读懂"。这是 LLM 的根本局限,没有机制能
  100% 保证一个模型"真读进去了"。我们能做的是把门禁前移、让漏读有据可查。
- 关卡2 自动派活会自动消耗额度。虽然你说烧钱不用管,但"自动"意味着失控风险更高,
  所以放在最后、且有并行度闸 + 前两关兜底才启用。
- 这套是"半自动流水线",不是无人值守工厂。我(或你)仍是最终裁判。
